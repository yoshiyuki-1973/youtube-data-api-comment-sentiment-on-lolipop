# 運用手順書

本書は、ロリポップ上で稼働する YouTube コメント感情分析システムの **運用・監視・障害対応** を整理する。  
デプロイ手順は [12_デプロイ手順書.md](12_デプロイ手順書.md) を参照すること。

---

## 1. 運用方針

- ロリポップ共有サーバー上で PHP アプリとして稼働する
- Docker は使用しない
- 分析結果は MySQL に 24 時間キャッシュする
- `video_id × comment_limit` は検索キーであり、UNIQUE 制約は置かない
- 同一キーの履歴は複数保持し、利用時は最新 `analyzed_at` を採用する

---

## 2. MySQL キャッシュ管理

### 2.1 キャッシュ仕様

| 項目 | 内容 |
|---|---|
| 有効期限 | 24 時間 |
| 検索キー | `video_id × comment_limit` |
| 履歴保持 | 同一キーを複数保持 |
| 採用ルール | 最新 `analyzed_at` を参照 |
| 関連テーブル | `videos` / `analysis_results` / `comments` |

### 2.2 キャッシュ削除

特定動画のキャッシュを削除する場合:

```sql
DELETE FROM videos WHERE video_id = 'VIDEO_ID_HERE';
```

全件削除する場合:

```sql
SET FOREIGN_KEY_CHECKS = 0;
TRUNCATE TABLE comments;
TRUNCATE TABLE analysis_results;
TRUNCATE TABLE videos;
SET FOREIGN_KEY_CHECKS = 1;
```

### 2.3 キャッシュ確認

```sql
SELECT video_id, title, fetched_at
FROM videos
ORDER BY fetched_at DESC;
```

```sql
SELECT v.title, ar.comment_limit, ar.total_count, ar.analyzed_at
FROM analysis_results ar
JOIN videos v ON v.video_id = ar.video_id
ORDER BY ar.analyzed_at DESC;
```

### 2.4 運用上の注意

- 同一 `video_id × comment_limit` でも履歴は複数レコード存在しうる
- 24 時間以内の再分析では、通常は最新キャッシュが返る
- 強制的に再分析したい場合はキャッシュ削除か、24 時間経過を待つ

## 3. API クォータ管理

### 3.1 YouTube Data API v3

| 項目 | 内容 |
|---|---|
| 日次クォータ | 10,000 units / 日 |
| リセット目安 | 太平洋時間 0:00 |
| 主な API | `videos.list`, `commentThreads.list` |

### 3.2 Grok API

- 利用量は xAI 側の契約・課金状況に従う
- コメント件数が多いほど消費量が増える
- モデル変更時は `config.php` の `GROK_MODEL` を見直す

### 3.3 負荷抑制の考え方

- UI は `10 / 30 / 50 / 100` の固定候補を採用する
- API は `1〜100` を受け付けるが、通常運用では UI 候補を優先する
- キャッシュを活用し、同一条件の短時間連打を避ける

## 4. エラー対応

### 4.1 まず確認する項目

1. `config.php` の API キーと DB 接続情報
2. ロリポップ側の PHP バージョン
3. `error_log()` に出ているメッセージ
4. MySQL に対象動画や分析結果が保存されているか

### 4.2 よくある症状

| 症状 | 確認ポイント |
|---|---|
| 500 エラー | `config.php`、API キー、`error_log()` |
| DB 接続エラー | `DB_HOST` / `DB_NAME` / `DB_USER` / `DB_PASS` |
| 分析結果が返らない | YouTube API 応答、Grok API 応答、入力 URL |
| 同じ結果しか出ない | キャッシュ有効期間内か、最新履歴を参照しているか |

### 4.3 ログ出力例

```php
error_log('[YT Sentiment] ' . $message);
error_log('[DEBUG] ' . print_r($variable, true));
```

## 5. セキュリティ

- `config.php` は Git に含めない
- `config.php` / `src/` / `sql/` は `.htaccess` で Web 非公開にする
- ディレクトリは `705`、公開ファイルは `604` を基本にする
- `config.php` は原則 `600` とし、動作しない場合のみ `604` に変更する
- `sql/*.sql` をサーバーへ残す場合は `600` とし、不要になったら削除する
- `777` や `666` は第三者が書き込めるため使用しない
- FTP パスワードは定期的に変更する
- API キーを README や公開資料へ書かない
